# Codex活用ノウハウ集（game63実践から抽出）

- 作成日: 2026-04-16
- 対象: game63 の実行ログ・運用文書・スプリント成果物
- 目的: Codex活用の質を上げるため、プロンプト群と進行方式を**評価→抽象化→再利用可能化**する。

---

## 0. 結論（先に厳しく）

game63の進め方は「文書化された運用」に成功している一方で、Codex活用としては次の弱点が残る。

1. **DoDは強いが、失敗基準（Fail Fast条件）が弱い。**
2. **イベント計測はあるが、改善判断の閾値運用が薄い。**
3. **プロンプトが逐次最適化寄りで、横断的な実験設計（A/B、比較対照）が不足。**
4. **最終QAは通っているが、回帰防止の自動化（最小テストセット）に未接続。**

つまり、
- 「進める力」は高い
- 「検証の厳しさ」と「再現可能な改善速度」はまだ伸びしろが大きい

---

## 1. 評価対象と前提

評価対象は次の3レイヤー。

1. **Prompt設計品質**
   - `NEXT_PROMPT_LOG.md` の Goal / Scope / Constraints / DoD / Next
2. **進行設計品質**
   - `PROJECT_MANAGER.md` の実行サイクル（方針→仕様→実装→検証→引継ぎ）
3. **成果物接続品質**
   - `outputs/` 群（討議→3案→マスタープラン→Sprint検証→Final QA）

---

## 2. Codex活用の評価（5段階）

## 2.1 Prompt設計

### 良い点
- 1プロンプト単位で目的が明確（Goal/Scope/DoDの分離）。
- 制約が本質保護に効いている（本質ループ維持、5歳UI、時間制約）。
- Nextを必ず付けるため、担当交代に強い。

### 課題（厳しめ）
- DoDは「達成したい状態」中心で、**「中止・巻き戻し条件」**が明示されていない。
- 失敗時の再計画テンプレートがなく、改善が担当者の技量依存になりやすい。
- KPIが文書上にある一方、プロンプト内に**「何セッション観測したら判定するか」**が弱い。

### 評点
- 設計明瞭性: 5/5
- 検証厳密性: 3/5
- 再計画容易性: 3/5

## 2.2 進行方式（PM運用）

### 良い点
- 参照優先順位が固定され、迷いを減らしている。
- 追記専用ログ + LATESTポインタが監査性を担保。
- スプリントごとに検証ドキュメントが残り、判断痕跡が追える。

### 課題（厳しめ）
- 「毎回読む」運用は強いが、**毎回測る**運用（同じ計測セット）が弱い。
- QAは最終で集中実施される傾向があり、途中での回帰検知が限定的。
- 保守モード移行後の受け入れ基準が、定量より定性寄り。

### 評点
- 実行再現性: 5/5
- 回帰耐性: 3/5
- 定量運用力: 3/5

## 2.3 成果物接続

### 良い点
- 討議→案比較→計画→実装→QAまで、ストーリーが切れていない。
- `game1` 本体から `game63` へ分離した判断は安全性が高い。

### 課題（厳しめ）
- 各Sprintの「期待差分」と「実測差分」が1枚で比較できるダッシュボード形式がない。
- プロンプト品質の自己評価（どの書き方が効いたか）が別管理されていない。

### 評点
- トレーサビリティ: 5/5
- 学習ループ速度: 3/5

## 3. 抽象化した再利用ノウハウ

## 3.1 まず使うべき「プロンプト骨格」

```md
# PROMPT-XXXX
- Goal:
- Scope:
- Non-Scope:  # やらないこと
- Constraints:
- DoD:
- Fail Fast:  # 中止条件・巻き戻し条件
- Metrics Plan: # 何を何回観測して判定するか
- Deliverables:
- Next:
```

**重要:** game63運用に足すべきは `Fail Fast` と `Metrics Plan`。

## 3.2 スプリントごとの最小実験設計

各Sprintを「実装」ではなく「仮説検証」として扱う。

- 仮説（H）: 例「段階ヒントで離脱率が下がる」
- 指標（M）: 例 `recall_fail` からのリトライ完遂率
- 閾値（T）: 例 +8%以上で採用、+3%未満は再設計
- 期間（W）: 例 最低10セッション
- 判定（D）: Adopt / Iterate / Revert

## 3.3 Codexへの指示粒度ルール

1. **設計タスク**: 仕様テンプレート埋めを先に要求。
2. **実装タスク**: 変更ファイル・非変更ファイルを明示。
3. **検証タスク**: 実行コマンドと期待出力をセットで指定。
4. **引継ぎタスク**: 次回Prompt雛形を同時生成させる。

---

## 4. アンチパターン（やると伸びない）

1. **「いい感じに改善して」型プロンプト**
   - 症状: 出力が毎回ぶれる。
   - 処方: Goal/DoD/制約を数値またはチェックボックス化。

2. **DoDのみでFail条件なし**
   - 症状: 失敗したのに前進扱いになる。
   - 処方: `Fail Fast` に「即巻き戻し条件」を書く。

3. **計測イベントを増やすだけで活用しない**
   - 症状: ログは多いが判断が主観化。
   - 処方: Sprintごとに「見る指標は3つまで」に絞る。

4. **最終QA一発勝負**
   - 症状: 後半で修正コストが跳ねる。
   - 処方: 各Sprint末に軽量回帰チェックを固定運用。

---

## 5. あなた（実行者）のCodex活用レベルを上げる練習メニュー

## レベル1: 仕様明確化（1週間）
- 毎回のPromptに `Non-Scope` と `Fail Fast` を追加。
- 実装着手前に「受け入れ条件チェック表」を先に作らせる。

## レベル2: 検証主導（2週間）
- 各Sprintで指標3つに絞り、閾値を事前宣言。
- 終了時に「想定 vs 実測」を1表で比較。

## レベル3: 学習高速化（継続）
- Promptテンプレートを固定し、書き方差分の効果を記録。
- 2スプリントごとに「プロンプト改善レビュー」を独立開催。

## 7. game63実績に対する最終講評（厳しめ総括）

- **良い:** 進行管理と文書化は上位水準。チーム開発で再現しやすい。
- **まだ弱い:** 「計測で意思決定する」運用の厳格さが不足。
- **次の一段:** Promptに Fail Fast + Metrics Plan を標準搭載し、
  「作るプロジェクト」から「学習速度を最適化するプロジェクト」へ移行する。

この転換ができると、Codexは単なる実装補助から、
**改善速度を上げる共同意思決定エンジン**になる。
