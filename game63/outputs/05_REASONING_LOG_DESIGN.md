# 思考ログ設計（後追い可能性を高める運用）

## 目的
- 設計変更の理由を、後から第三者が追跡できる状態で残す。
- ただし、冗長な逐語思考ではなく、**意思決定に必要な要約ログ**を残す。

## ログの原則
1. **結論先行**: 何を決めたかを最初に記載
2. **根拠明示**: 参照元（本質定義、ユーザー要求、テスト結果）を記載
3. **代替案比較**: 少なくとも却下案1つと却下理由を記録
4. **再現可能性**: 次に誰が見ても同じ判断に近づける粒度で書く
5. **非保存対象**: 逐語的な内省や未整理メモは保存しない

## 推奨ログ項目（1イベント1レコード）
- `timestamp`
- `phase`（要件整理 / 発散 / 収束 / 決定 / 検証）
- `question`（何を判断したか）
- `decision`（採用した結論）
- `alternatives`（比較候補）
- `rationale`（採用理由）
- `evidence`（根拠ファイルやデータ）
- `impact`（影響範囲）
- `next_action`
- `owner`

## 保存フォーマット

```yaml
- timestamp: 2026-04-15T10:00:00Z
  phase: 決定
  question: 先行実装案はどれか
  decision: 案1をMVP採用
  alternatives:
    - 案2
    - 案3
  rationale:
    - 実装容易性が最も高い
    - 5歳の即時成功体験を作りやすい
  evidence:
    - GAME_ESSENCE.md
    - 02_PERSONA_PANEL_DISCUSSION.md
  impact:
    - UI: チャンク表示追加
    - ロジック: 難易度テーブル追加
  next_action: MVP仕様の詳細化
  owner: director
```

## ログ運用ルール
- 重要判断ごとに1件、1日最大10件程度
- リリース時に「主要判断サマリ」を別紙で作成
- ログ更新責任者はディレクター
